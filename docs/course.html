What This App Does

A Hotel in Your Browser

Hotel Porec is a small hotel on the Croatian coast. This web application replaces paper ledgers and spreadsheets with a live digital dashboard that the whole staff shares.

The heart of the app is the Front Desk timeline — a scrollable 14-day grid where rows are rooms and columns are dates. A receptionist can see every booking at a glance, drag to create new ones, and check guests in or out with a click.

How It Starts Up

When a receptionist opens the app, there's a precise startup order — like turning the key in a car's ignition. Each system must boot before the next one can begin.

1
Language detection

The app detects whether the user speaks Croatian, German, or English and loads the right translations.

2
Data cache initializes

TanStack Query sets up its in-memory cache — a temporary scratchpad that holds recently fetched data so the UI feels instant.

3
Authentication resolves

The app checks who you are via Supabase Auth. Until this finishes, the whole UI is paused on a loading spinner.

4
Router renders

Only now does the router kick in and show the right page. This ordering prevents users from briefly seeing a “not logged in” flash.

The Data Registry

Every piece of data the app fetches gets a label — a query key. This is the central registry that prevents data from getting mixed up.

CODE 
export const queryKeys = {
  rooms: {
    all: () => ['rooms'] as const,
    detail: (id) => ['rooms', id] as const,
  },
  guests: {
    all: () => ['guests'] as const,
    search: (query) => ['guests', 'search', query],
  },
  reservations: {
    all: () => ['reservations'] as const,
  },
};
PLAIN ENGLISH

Create a central registry of all data labels the app uses.

For rooms: "all rooms" gets the label ['rooms'].

A specific room gets ['rooms', '42'] — the ID makes it unique.

Same pattern for guests: "all guests" is ['guests'].

Searching guests by name gets its own label: ['guests', 'search', 'smith'].

And reservations follow the same convention.

This means when a reservation changes, the system knows exactly which cached data to refresh — without touching unrelated caches.

💡
Key Insight: Single Source of Truth

This "one registry for all labels" pattern prevents the #1 caching bug: two parts of the app fetching the same data with different labels, resulting in stale data that never refreshes. If you ever build a data-heavy app, this is the first thing to set up.

A Data Hook in Action

This is one of the most important files in the codebase. It defines how reservations are fetched from the database and what shape the data takes.

CODE 
function buildReservationsQuery() {
  return supabase
    .from('reservations')
    .select(`*,
      reservation_statuses!status_id(code),
      booking_sources!booking_source_id(code),
      guests!guest_id(id, first_name, last_name,
        full_name, email, phone, nationality,
        has_pets, is_vip, vip_level),
      labels!label_id(id, name, color, bg_color)`)
    .order('check_in_date');
}

export type Reservation =
  QueryData<ReturnType<
    typeof buildReservationsQuery
  >>[number];
PLAIN ENGLISH

Build a request to the database that says:

"Go to the reservations table..."

"...and give me everything, plus follow these links to related tables:"

• The reservation's status (confirmed, checked-in, etc.)

• Where it was booked from (Booking.com, direct, etc.)

• The guest's name, contact info, and VIP status

• Any label attached (VIP, corporate, etc.)

"Sort everything by check-in date."

Then automatically create a TypeScript type from whatever the database returns — so the type updates itself when the database schema changes.

The "Assume Success" Trick

When a receptionist changes a reservation's status from "confirmed" to "checked-in," the UI updates instantly — before the database even receives the request. This is called an optimistic update.

CODE 
onMutate: async ({ id, status }) => {
  await queryClient.cancelQueries({
    queryKey: queryKeys.reservations.all()
  });
  const previous =
    queryClient.getQueryData(queryKeys.reservations.all());
  queryClient.setQueryData(queryKeys.reservations.all(),
    (old = []) => old.map((r) =>
      r.id === id
        ? { ...r, reservation_statuses: { code: status } }
        : r
    )
  );
  return { previous };
},
onError: (_err, _vars, context) => {
  if (context?.previous)
    queryClient.setQueryData(
      queryKeys.reservations.all(), context.previous);
},
PLAIN ENGLISH

Before sending to server:

Stop any in-flight fetches for reservations (so they don't overwrite our optimistic change).

Take a snapshot of the current reservation list — this is our safety net.

Immediately update the cached list: find the reservation by ID and swap its status to the new value.

The UI re-renders instantly. The user sees "checked-in" right away.

Return the snapshot so we can use it later if things go wrong.

If the server says "error":

Restore the snapshot we saved. The UI reverts as if nothing happened.

💡
Key Insight: Snapshot & Rollback

This "save a backup, then change optimistically, then rollback on failure" pattern appears everywhere in software — from database transactions to undo/redo in text editors. It's the foundation of making things feel instant while keeping them safe.

The Wiring Behind Realtime

This tiny file is the bridge between Supabase's live data stream and the app's cache. When the database changes, this code tells TanStack Query to throw away its stale copy and fetch fresh data.

CODE 
export function useRealtimeSync() {
  const queryClient = useQueryClient();

  useEffect(() => {
    const unsubscribe =
      realtimeService.subscribeToHotelTimeline(
        // Reservations changed
        () => queryClient.invalidateQueries({
          queryKey: queryKeys.reservations.all()
        }),
        // Rooms changed
        () => queryClient.invalidateQueries({
          queryKey: queryKeys.rooms.all()
        }),
        // Guests changed
        () => queryClient.invalidateQueries({
          queryKey: queryKeys.guests.all()
        }),
      );
    return () => unsubscribe();
  }, [queryClient]);
}
PLAIN ENGLISH

Create a hook that keeps the app in sync with the database.

Get a reference to the data cache manager.

When this hook is first mounted, open a live WebSocket connection to Supabase.

Whenever a reservation is inserted, updated, or deleted in the database — mark the reservations cache as stale (triggering a refetch).

Same for rooms.

Same for guests.

When the component unmounts, close the connection to avoid memory leaks.

Season Detection in Code

The pricing engine's first job is to figure out which season a given date falls in. Here's the surprisingly simple function that does it:

CODE 
getSeasonalPeriod(date: Date): SeasonalPeriod {
  const month = date.getMonth() + 1;

  if (month <= 4 || month === 12) return 'A';
  if (month === 5 || month >= 10) return 'B';
  if (month === 6 || month === 9) return 'C';
  if (month >= 7 && month <= 8) return 'D';

  return 'A';
}
PLAIN ENGLISH

Given a date, figure out which pricing season it falls into.

Get the month number (JavaScript months start at 0, so we add 1 to get Jan=1, Dec=12).

Jan through Apr, or December? That's the low season — return A.

May, or Oct through Nov? Shoulder season — return B.

June or September? Early/late summer — return C.

July or August? Peak summer on the Croatian coast — return D (most expensive).

Fallback to A, just in case. (This line should never actually run.)

When a Stay Crosses Season Boundaries

Here's the clever part: a guest might check in during one season and check out during another. The engine splits the stay into season blocks and charges each block at its own rate.

C
Jun 29 – Jun 30 (2 nights)

Season C rate applies. Early summer pricing.

D
Jul 1 – Jul 3 (3 nights)

Season D rate kicks in. Peak summer pricing. The same room just got more expensive.

Example: A 5-night stay from Jun 29 to Jul 4 is split into two blocks, each billed at the correct seasonal rate.

📚
The Full Charge Breakdown

Beyond accommodation, the pricing engine also calculates: tourism tax (EUR 1.60/person in summer, EUR 1.10 in winter), child discounts (free under 3, 50% off ages 3-6, 20% off ages 7-13), parking (EUR 7/night), pets (EUR 20/night), and a +20% short-stay surcharge for stays under 3 nights.

Safety Nets

What happens when two receptionists try to book the same room, or the database goes down mid-operation? The system has layers of protection.

The Collision Detector

Imagine an air traffic controller who checks every incoming flight against all other scheduled landings. The ConflictDetectionService does the same thing for bookings — before any reservation is saved, it's checked against every existing booking for that room.

CODE 
export interface ConflictResult {
  hasConflict: boolean;
  conflicts: BookingConflict[];
  warnings: BookingWarning[];
  suggestions: BookingSuggestion[];
}

export interface BookingConflict {
  type: 'room_occupied'
    | 'overlapping_reservation'
    | 'maintenance_period'
    | 'room_unavailable';
  severity: 'error' | 'warning';
  message: string;
  suggestedAlternatives?: Room[];
}
PLAIN ENGLISH

When checking a booking, the system returns a structured report:

hasConflict — a simple yes/no answer: can this booking proceed?

conflicts — the specific problems found (room occupied, dates overlap, maintenance, room closed).

warnings — not blockers, but things to watch out for (short stay, peak period, etc.).

suggestions — alternative rooms or dates if there's a conflict.

Each conflict has a type (what kind of problem), a severity (block or just warn), a human-readable message, and possibly a list of alternative rooms.

Defense in Depth

The app has multiple layers of protection, each catching different kinds of failures — like safety nets at different heights below a trapeze artist.

1
Form Validation (Zod schemas)

Before anything reaches the server, the form checks locally: Are dates valid? Is a guest selected? Is the room chosen? Catches typos and missing fields.

2
Conflict Detection (Service layer)

The ConflictDetectionService checks business rules: date overlaps, room availability, maintenance periods. Prevents impossible bookings.

3
Optimistic Rollback (Cache layer)

If the server rejects a mutation, the onError handler restores the cache snapshot. The UI reverts seamlessly.

4
Error Boundary (Last resort)

If something completely unexpected happens, the ErrorBoundary component catches the crash and shows a recovery screen instead of a blank page.

💡
Key Insight: Layers of Protection

"Defense in depth" is a fundamental principle in both software and security. Each layer assumes the previous one might fail. Form validation catches user errors, conflict detection catches business rule violations, rollback catches server failures, and the error boundary catches everything else. No single layer needs to be perfect.

Key Design Decisions

Every app is shaped by the tradeoffs its builders made. Here are the most important ones — and why they matter when you're directing an AI to extend the app.

🔍

TQ as the Cache

No Redux store. TanStack Query is both the fetcher and the cache. This means less code to maintain, but every piece of data must go through a hook.

📂

No Backend Server

Supabase replaces a traditional backend. Components talk to the database directly through the Supabase SDK. Faster to build, but business logic lives in the frontend.

⚖

Services for Rules

Pricing, conflicts, and booking logic live in standalone service files — not in components. This makes the logic testable and reusable across different UI flows.

🌐

Three Languages

Every string goes through i18n with Croatian, German, and English translations. Adding a fourth language means adding one JSON file — no code changes.
